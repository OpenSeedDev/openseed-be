# VS-027 피드백 채택과 기여 보상

## 사용자 결과

아이디어 게시자는 이유나 수정 방향을 입력하지 않고 활성 피드백을 한 번 채택하며, 피드백 작성자는 기여자로 기록되고 100P 활동 보상과 성장 타임라인 기록을 확인할 수 있다.

## 근거와 선행 조건

- 백로그: VS-027, 선행 VS-024, `feedback`·`wallet` resource lock
- 실제 Merge 증거: VS-024 PR #53이 `335bd366835a6474f2901655b816891e6e8b4749`로 Merge됨
- 기능 명세 FB-03~07·POINT-06: 작성자만 한 번 채택, 입력 본문 없음, Contribution·100P·타임라인 원자 처리, 동일 기여자·아이디어·Asia/Seoul 정책 날짜당 한 번만 지급
- `PointRewardService`의 일일 활동 300P와 지갑 2,000P 상한·전액 소멸 원장 정책을 재사용한다.

## 포함 범위

- `POST /api/v1/feedbacks/{feedbackId}/accept`
- 아이디어 작성자 권한, 활성·미채택 피드백 검증
- 피드백 `accepted_at`, Contribution, `FEEDBACK_ACCEPTED` 타임라인, 100P 원장을 하나의 트랜잭션으로 처리
- 같은 기여자·아이디어의 같은 정책 날짜 두 번째 채택은 Contribution과 타임라인은 생성하되 100P 전액 소멸 원장을 기록
- 같은 피드백의 반복·동시 채택 방지
- Point 보상 실패 시 채택·Contribution·타임라인 전체 롤백
- 새 UTC Flyway Migration, JPA 매핑, OpenAPI와 ERD 갱신

## 제외 범위

- 채택 취소
- 채택 이유·수정 방향 입력
- AI 보강·자동 버전 연결
- 기여자 Top 5와 마이페이지 기여 목록
- 랭킹 점수 계산과 별도 지표 집계

## API와 데이터 영향

- 요청 본문은 없다.
- 응답은 피드백·아이디어·Contribution·기여자 ID, 채택 시각과 보상 원래·실지급·소멸 금액을 반환한다.
- 인증 실패는 `401 INVALID_ACCESS_TOKEN`, 존재하지 않거나 삭제된 피드백 및 작성자 외 요청은 `404 FEEDBACK_NOT_FOUND`, 이미 채택된 피드백은 `409 FEEDBACK_ALREADY_ACCEPTED`다.
- `contributions`는 아이디어·기여자·원본 피드백·생성 시각을 저장하고 원본 피드백당 하나를 DB 유일 제약으로 보장한다.
- `point_ledgers.reward_scope_id`에 아이디어 ID를 저장해 기여자·아이디어·정책 날짜별 지급 한도를 직렬화된 지갑 잠금 안에서 계산한다.
- 타임라인에 `FEEDBACK_ACCEPTED` 유형과 원본 피드백 `source_id`를 저장한다.

## 인수 조건과 테스트 목록

- [ ] 아이디어 작성자가 활성 피드백을 채택하면 acceptedAt·Contribution·100P·타임라인이 함께 생성된다.
- [ ] 요청에는 채택 이유·수정 방향 본문이 필요하지 않다.
- [ ] 인증 누락·위조 토큰은 401이다.
- [ ] 작성자 외 요청, 삭제·없는 피드백은 존재를 숨겨 404다.
- [ ] 이미 채택된 피드백의 반복 요청은 409이며 추가 기록이나 보상이 없다.
- [ ] 같은 피드백의 동시 채택에서도 성공은 한 번이고 Contribution·원장·타임라인은 각각 한 건이다.
- [ ] 같은 기여자·아이디어·정책 날짜의 두 번째 채택은 기여 기록을 만들되 100P를 전액 소멸한다.
- [ ] 다른 아이디어 또는 다음 정책 날짜의 채택은 다시 100P 지급 대상이다.
- [ ] 지갑 상한·일일 활동 상한 초과분은 기존 활동 보상 정책대로 소멸한다.
- [ ] Point 처리 실패 시 채택·Contribution·타임라인도 롤백된다.
- [ ] OpenAPI에 Bearer 인증과 200·401·404·409 계약이 노출된다.
- [ ] PostgreSQL 제약, 집중 테스트와 전체 회귀 테스트가 통과한다.
