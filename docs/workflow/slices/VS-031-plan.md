# VS-031 활동 Point 지급 정책

## 사용자 결과

활동 보상을 받을 사용자는 일일 활동 한도와 지갑 상한 안에서 지급 가능한 Point를 받고, 초과분은 대기 잔액으로 넘기지 않고 즉시 소멸한 내역까지 원장에서 확인할 수 있다.

## 근거와 선행 조건

- 백로그: VS-031, 선행 VS-030, `wallet`·`feedback` resource lock
- 실제 Merge 증거: VS-030 PR #30이 `9454302d4cd93f7df13a03f5737dd98d452c4b6c`로 2026-07-21T06:28:10Z Merge됨
- VS-030은 현재 지갑과 Append-only 원장 조회, 원장 UPDATE·DELETE 차단을 제공한다.
- 정책: 가입 보너스를 제외한 활동 보상은 Asia/Seoul 하루 300P, 지갑은 2,000P로 제한한다.

## 포함 범위

- 공개·반공개 아이디어 게시 50P, 구조화 피드백 20P, 피드백 채택 기여 100P 활동 보상 유형
- 사용자별 지갑 행 잠금 후 일일 지급액과 지갑 여유분을 함께 적용하는 원자적 지급
- `sourceType + sourceId` 중복 지급 키
- 원래 보상, 실제 지급, 초과 소멸, 지급 후 잔액을 Append-only 원장에 항상 기록
- Asia/Seoul 00:00~23:59 정책 날짜 경계
- DB 유형·금액·중복 제약과 ERD 갱신

## 제외 범위

- 게시·피드백·채택 기능의 API와 엔티티 생성: VS-010, VS-024, VS-027
- 각 활동의 횟수 제한: 게시 하루 2회, 피드백 하루 5회, 동일 기여자·아이디어 채택 하루 1회는 해당 활동 슬라이스에서 검증한다.
- 일일 첫 접속 보상: VS-056
- Point 차감·Unit 구매·회수: VS-032~036
- Point 만료·양도·현금화·관리자 조정

## 애플리케이션과 데이터 영향

- 후속 활동 슬라이스가 트랜잭션 안에서 호출할 `PointRewardService`와 고정 `ActivityReward` 계약을 제공한다.
- `point_ledgers`에 정책 날짜와 중복 지급 키를 저장하고, 활동 보상 출처 유형을 확장한다.
- 보상 시도는 지급액이 0이어도 원래 금액과 전액 소멸을 원장에 남긴다.
- 공개 HTTP API는 추가하지 않으므로 OpenAPI 경로 변경은 없다.

## 인수 조건과 테스트 목록

- [ ] 게시 50P, 피드백 20P, 채택 100P가 고정 정책대로 지급된다.
- [ ] 가입 보너스는 일일 활동 합계에서 제외된다.
- [ ] 당일 활동 지급 합계가 300P를 넘으면 나머지만 소멸한다.
- [ ] 지갑이 1,950P일 때 100P 보상은 50P 지급·50P 소멸한다.
- [ ] 일일 한도와 지갑 상한을 함께 적용할 때 더 작은 지급 가능 금액을 사용한다.
- [ ] 소멸분은 `pending_recovery_balance`로 이동하지 않는다.
- [ ] 같은 출처 키 재호출은 새 지급·원장을 만들지 않는다.
- [ ] 지갑 행 잠금과 DB 유일 제약으로 동시 요청도 한 번만 지급된다.
- [ ] Asia/Seoul 자정 전후 보상은 서로 다른 정책 날짜로 집계된다.
- [ ] 원장은 원래·지급·소멸·지급 후 잔액과 출처·정책 날짜를 보존한다.
- [ ] 집중 테스트와 `./gradlew clean test`가 통과한다.
